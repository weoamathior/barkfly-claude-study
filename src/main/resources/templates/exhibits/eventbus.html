Global Window Event Bus

tagline

Why enterprises do this

Apps on the same page need to react to each other ("an order was selected → refresh the invoice"). window.dispatchEvent is right there, needs no wiring, and works across teams' bundles. So it becomes the integration bus by accident.

Why it hurts

  • Event names are ad-hoc strings — two teams pick different names for the same thing (silent misses) or the same name for different things (collisions).
  • No ownership, no docs — nobody knows who listens.
  • Duplicate listeners fire the same side effect repeatedly (duplicate invoices!).
  • Anonymous handlers can't be removed → leaks and zombie reactions.

BAD — Raw window events, ad-hoc names, no dedup

Toggles & reset

Scripted walkthrough

  1. BAD: Click Add invoice listener three times, then Publish "orderSelected". Watch 3 invoices generated for one selection.
  2. BAD: Click Publish "order_selected" (the other team's name). Nothing happens — a silent miss. Then try Remove listener: it can't, the handler was anonymous.
  3. BETTER: Click Subscribe invoice handler twice — the second is rejected with a duplicate-listener warning. Publish once → exactly one invoice.
  4. BETTER: Click Publish invalid payload → rejected by schema validation. Then Unsubscribe → it actually unsubscribes.
