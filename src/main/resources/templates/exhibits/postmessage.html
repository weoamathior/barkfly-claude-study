postMessage Communication

tagline

Why enterprises do this

The portal and the embedded app live on different origins, so they can't share memory or call each other directly. postMessage is the one sanctioned channel — so everything gets crammed through it, usually without a contract.

Why it hurts

  • postMessage(payload, '*') and a handler that trusts any origin → spoofable.
  • Loosely-typed payloads, no version, no schema validation.
  • No correlation ID → can't match a reply to its request.
  • No timeout → a silent child hangs the feature forever.

Toggles

These affect the better request so you can watch the timeout and schema validation do their jobs. The bad request has no such defenses.

Scripted walkthrough

  1. Click Request order (bad) and Request order (better). Both work — note the better one carries a correlationId in the log.
  2. Click Inject spoofed reply. The bad output happily shows the attacker's fake order; the better handler logs REJECTED_BAD_ORIGIN and ignores it.
  3. Tick Simulate dropped response, click Request order (better): watch it hit a timeout instead of hanging forever. The bad request would just wait silently.
  4. Tick Simulate malformed reply, click Request order (better): the schema check rejects the corrupted payload.
