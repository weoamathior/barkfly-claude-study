Shared localStorage / sessionStorage

tagline

Why enterprises do this

Two apps on the same origin can both reach localStorage, so it becomes a tempting "shared bus": the account picker writes the selected account, the orders app reads it. No backend changes, no API. Ship it.

Why it hurts

  • It's an implicit contract: one app's shape change silently breaks the other.
  • No version, no expiry → stale data shows the wrong account's orders.
  • No defensive parsing → one bad value throws and breaks the reader.
  • Logout clears one app's keys, not the others' → ghost session.

Below, "Account Picker" and "Orders Viewer" stand in for two separate apps that happen to share this origin's localStorage.

BAD — Raw key, assumed shape, no expiry, partial logout

🧾 Account Picker
localStorage["selectedAccount"] = (unset)
📦 Orders Viewer
No orders loaded.

BETTER — Versioned envelope, expiry, defensive parse, clear-all logout

🧾 Account Picker
storage = (unset)
📦 Orders Viewer
No orders loaded.
Envelope: { "version": 2, "owner": "account-picker", "ts": …, "expiresAt": …, "data": { "id": "…", "name": "…" } }

Scripted walkthrough

  1. BAD: Select Mission Pharmacy, then Refresh from storage — orders load. Now click Logout (bad) and Refresh again: the orders are still there (ghost session).
  2. BAD: Click Write legacy v1 shape, then Refresh: the reader assumed .id and gets the wrong/empty result.
  3. BAD: Click Corrupt the value, then Refresh: the reader throws and the panel breaks.
  4. BETTER: Repeat each step. Expired/corrupt/old-version values are detected and reported; Logout (better) clears everything.
