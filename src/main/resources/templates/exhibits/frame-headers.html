Iframe Security Headers / Clickjacking

tagline

Real vs simulated: the frame-ancestors / X-Frame-Options enforcement below is real — your browser actually refuses to render the blocked frame. The "attacker overlay" is a simulation to illustrate the clickjacking idea safely.

Why this tension exists

Integration wants apps to embed each other freely. Security wants to stop hostile pages from framing your app and tricking users into clicks (clickjacking). Those goals collide in the framing headers.

Why it hurts

  • No framing policy → any site can embed your app and overlay a deceptive UI.
  • Too strict (DENY) → your own portal can't embed it either.
  • The fix is an explicit allowlist, which integration teams often don't know to ask for.

BAD — Unprotected page + attacker overlay (simulated clickjack)

In a real clickjack the victim app is the invisible top layer and the decoy is underneath — the user thinks they're claiming a prize but actually clicks "Confirm transfer". Here the overlay is shown semi-transparent so you can see the trick.

BETTER — Explicit framing policy (real enforcement)

The allowlisted page embeds fine because the portal origin is explicitly permitted. The denied page is refused by the browser — watch the frame stay blank and check the devtools console for the CSP/X-Frame-Options error.

Scripted walkthrough

  1. BAD: Embed unprotected bank page — it frames with no complaint. Tick Show attacker overlay to see how a hostile page would deceive the user.
  2. BETTER: Embed allowlisted page — it loads, because frame-ancestors names the portal origin.
  3. BETTER: Embed denied page — the frame stays blank; the browser refused it. Open devtools console to see the real CSP refusal.
  4. Reflect: the safe answer isn't "no framing" or "frame anything" — it's an explicit allowlist agreed between the apps.
