Zoo / Exhibit #10

Runtime-Origin Confusion

tagline

Why enterprises do this

On the developer's laptop, everything runs on localhost:8080. So code reaches for window.location.origin as "the other app's origin" and it just works — until staging and prod split the apps onto different hosts.

Why it hurts

  • "Works on my machine" — same-origin assumptions pass every local test.
  • In prod the apps are on different origins: postMessage targetOrigin no longer matches and messages are silently dropped.
  • Origin checks written as === location.origin reject the real partner.

Runtime diagnostics

BAD — Assumes the child shares the portal's origin

No response yet.

It posts to window.location.origin (the portal). The child is on a different origin, so the browser drops the message — no error, no reply.

BETTER — Config-driven origin + allowlist + mismatch diagnostics

No response yet.

Origins come from environment config, replies are checked against an explicit allowlist, and a mismatch produces a clear diagnostic instead of a silent drop.

Reset

Scripted walkthrough

  1. Read the diagnostics: the portal and child origins differ here (prod-like), so same-origin assumptions will break.
  2. BAD: Request order (targetOrigin = location.origin) → no reply arrives; the browser silently dropped the message because the target origin didn't match the child.
  3. BETTER: Request order (configured child origin) → it works, because the origin came from config, not an assumption.
  4. BETTER: Simulate reply from an unexpected origin → a clear "origin not in allowlist" diagnostic, not a silent failure or a blind accept.
