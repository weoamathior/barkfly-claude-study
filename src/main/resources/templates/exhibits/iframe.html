Portal Shell + Iframe Legacy App

tagline

Why enterprises do this

The portal team can't rewrite the 2014 orders app, so they wrap it in an iframe and ship "integration" in an afternoon. Two apps, one URL bar, everyone's happy at the demo.

Why it hurts

  • Hardcoded child URL; no load or error handling — failures are silent blank frames.
  • Fixed height clips content and creates a scroll-trap.
  • No visible ownership boundary — users don't know they crossed into another app.
  • Back button and "open in new tab" behave unpredictably.

BAD — Hardcoded iframe, no wrapper

Hardcoded src, fixed 120px height, no load/error/latency handling.

Nothing loaded.

BETTER — Wrapper with boundary, load/error/timeout states

Visible app boundary, spinner while loading, waits for a CHILD_READY handshake from the allowed origin, and shows a real error state on failure or timeout.

Integration contract: child must post {type:'CHILD_READY', source:'legacy-orders-app'} from :8081 within 5s of load.

Toggles

Pick a scenario, then click a Load button above. Watch how the two wrappers react differently to the same failure.

Scripted walkthrough

  1. Select Backend 500, then click Load legacy app (bad wrapper). Observe: a blank/broken frame, no explanation. The log shows the wrapper set src and then… nothing.
  2. Click Load legacy app (better wrapper) with the same scenario. Observe: spinner, then a clear error state when the handshake times out.
  3. Switch to Slow load and load both. The bad frame is blank during the delay; the better frame shows a spinner.
  4. Switch to Normal: both load, but only the better one shows the app boundary and isn't clipped at 120px.

Note: the child app is served from a genuinely different origin (:8081), so the CHILD_READY origin check in the better wrapper is real.
