Token Passing Anti-pattern

tagline

Why enterprises do this

The child app needs to know who you are, so the portal just hands it the token — in the URL, in localStorage, wherever's easy. And since the token says role=ADMIN, the child trusts it.

Why it hurts

  • Tokens in URLs leak into history, logs, and referrers.
  • Tokens in localStorage are readable by any script (XSS = game over).
  • Tokens logged to the console end up in screen-shares and bug reports.
  • Trusting a client-supplied role lets anyone become ADMIN by editing a value.

BAD — Token in URL + localStorage + console, client-trusted role

No token in the URL or localStorage. The child fetches a minimal context, and the role comes from the server's view of the user — editing client values changes nothing.

Scripted walkthrough

  1. BAD: Login as Dana (Account Manager). Note the token sitting in the URL, in localStorage, and printed to the console/log.
  2. BAD: Tick Tamper: force role=ADMIN, Open child app, then do ADMIN action: the child trusts the URL and grants admin to a non-admin. 🔓
  3. BETTER: Login as Dana via BFF — only an opaque session id exists. Load child context shows role ACCOUNT_MANAGER (from the server).
  4. BETTER: do ADMIN action → the server returns 403. Switch to Priya (Admin) and it's allowed. The client can't escalate.
